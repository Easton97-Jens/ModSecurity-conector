# New Connector Onboarding

This document defines the evidence gates for future ModSecurity connector work.
It is intentionally stricter than a scaffold checklist: a connector must not be
promoted by intent, naming, or expected status changes. Promotion requires files,
runtime artifacts, and generated evidence.

## Scope

New connector planning is roadmap-only until a runtime proof exists. Roadmap-only
material may describe architecture options, risks, and proof steps, but it must
not create runtime PASS/FAIL values, full-matrix rows, merge-readiness signals,
or production status.

The current roadmap report is generated by `ci/generate-connector-roadmap.py` and
written under `reports/testing/generated/manifest/connector-roadmap.generated.*`.
It is registered as `roadmap_only` and does not require runtime or full-matrix
inputs.

## Connector Lifecycle

| Stage | Meaning | Required Evidence | Not Allowed Claims |
| --- | --- | --- | --- |
| planned | Candidate is tracked, but no connector directory is required. | Roadmap entry, risks, and first proof step. | Runtime support, CRS support, verified-case readiness. |
| skeleton | Directory or starter exists, but runtime behavior is not proven. | README, metadata/origin notes, build starter or self-test, explicit non-runtime disclaimer. | Runtime PASS/FAIL, production readiness, full-matrix eligibility. |
| buildable | Starter or adapter component compiles reproducibly. | Build command, output outside checkout, source map, lint/governance pass. | Traffic handling or ModSecurity semantics without runtime evidence. |
| runtime-startable | Server/proxy and sidecar can start locally. | Minimal config, start/stop script, process logs, deterministic cleanup. | Verified blocking or CRS support without result.json/logs. |
| verified-case-ready | One targeted real runtime case can run. | result.json, case-run files, access/error logs, decision/audit evidence if applicable. | Full-matrix readiness or broad phase coverage. |
| full-matrix-candidate | Full-matrix jobs are technically schedulable. | Runtime producer, matrix job entries, report integration, known-limitation notes. | Full-matrix PASS or merge-readiness contribution before evidence. |
| production-verified | Complete verified evidence pipeline has passed. | Verified runtime evidence, full matrix, governance, lint, quick-check, report layout, merge-readiness PASS. | Any production claim if generated evidence is blocked, stale, or missing. |
| covered-by-existing-connector | Runtime is intentionally handled as a variant of an existing connector. | Decision record naming owning connector and compatibility-smoke path. | Separate connector identity, separate generated reports, separate full matrix. |
| blocked | External, licensing, architecture, or evidence issue prevents progress. | Blocker description and unblocking proof. | Status promotion before blocker resolution. |

## Acceptance Criteria

A new connector may not move toward Full-Matrix until it can prove at least:

- `connectors/<name>/README.md`
- build/start/stop or run script
- minimal runtime configuration
- verified-case support
- `result.json`
- access/error logs or equivalent
- audit evidence when supported
- decision evidence when applicable
- capability notes
- request-blocking smoke
- request-body smoke or a documented not-supported reason
- clean `make report-governance`, `make lint`, and `make quick-check`

## Evidence Rules

- Do not manually patch generated reports.
- Do not change expected statuses to make a new connector look green.
- Do not invent PASS/FAIL values or full-matrix rows for non-implemented connectors.
- Do not commit blocked generated reports from a local environment missing verified runtime inputs.
- Store runtime artifacts under the verified runtime root, not inside the checkout.
- Keep new connector reports roadmap-only until real runtime evidence exists.

## Envoy First Proof

Envoy is the next recommended connector because the repository already contains
an Envoy starter and Envoy has realistic HTTP filter control points for a
sidecar-based proof. The first step is not a full connector. It is a targeted
runtime smoke that proves a minimal request-blocking path.

Recommended proof:

- Minimal Envoy `ext_proc` or `ext_authz` runtime smoke.
- Envoy starts locally with a deterministic minimal config.
- A simple upstream responds through Envoy.
- A ModSecurity-like decision service or sidecar returns a deny decision.
- A smoke case such as `envoy_request_blocking_smoke` returns HTTP 403.
- `result.json`, Envoy logs, decision-service logs, and case-run files are
  written under `$VERIFIED_RUN_ROOT/envoy-smoke/`.

Envoy first-proof non-goals:

- No CRS support.
- No MRTS support.
- No full matrix.
- No response-body support claim.
- No `production_verified` claim.
- No merge-readiness impact.

## LiteSpeed Planned Proof

LiteSpeed is a planned candidate only. OpenLiteSpeed is likely the first variant
to study because it is more likely to be CI-friendly than LiteSpeed Enterprise,
but packaging, licensing, download automation, and ModSecurity/CRS compatibility
must be proven before any connector directory is added.

First proof: OpenLiteSpeed install/start proof plus one CRS/request-blocking
smoke, if automation and licensing allow it.

## OpenResty Decision

OpenResty is covered by the existing NGINX connector because it is NGINX-based.
It must not become a separate connector at this stage. A future compatibility
smoke may be added as an NGINX runtime variant, but it must not create separate
Full-Matrix rows or separate generated connector reports.

## What Not To Claim Before Full-Matrix Evidence

Before full-matrix evidence exists, do not claim:

- production-verified status
- merge-readiness contribution
- full-matrix PASS/FAIL counts
- CRS support across variants
- MRTS support
- response-body blocking support
- phase coverage parity with Apache, NGINX, or HAProxy
- runtime capability support without `result.json` and logs
